Создание собственного сервиса в Linux — это база для любого админа и DevOps. В современном Arch Linux за управление службами отвечает система инициализации systemd.

Давай пошагово оформим наш Python HTTP-сервер как настоящий системный демон.

Шаг 1. Готовим скрипт и рабочую директорию
Создадим отдельную папку и простой Python-скрипт, чтобы не запускать сервис от имени root и не хранить его где попало.

Создаем папку для сервиса:

Bash
mkdir -p ~/my_service
cd ~/my_service
mkdir -p (make directory) — флаг -p (parents) создает всю цепочку папок, если их нет, и не выдает ошибку, если папка уже существует.

cd (change directory) — переходим в созданную папку.

Создаем тестовый HTML-файл, который будет отдавать сервер:

Bash
echo "Hello from Systemd Service!" > index.html
Шаг 2. Создаем Юнит-файл (Unit file) для systemd
Все службы в systemd описываются через конфигурационные файлы с расширением .service. Системные юниты, созданные пользователем, принято класть в директорию /etc/systemd/system/.

Создаем и открываем файл сервиса:

Bash
sudo nano /etc/systemd/system/myserver.service
Вставляем туда следующий конфиг:

Ini, TOML
[Unit]
Description=My Custom Python HTTP Server
After=network.target

[Service]
Type=simple
User=sasha
WorkingDirectory=/home/sasha/my_service
ExecStart=/usr/bin/python3 -m http.server 8080

# Автоматический перезапуск при падении
Restart=always
RestartSec=3

# Ограничения ресурсов (cgroups v2)
MemoryMax=100M
MemoryHigh=80M

[Install]
WantedBy=multi-user.target
Разбор конфига по косточкам:
Секция [Unit] (Метаданные и зависимости)
Description — понятное человеческое описание сервиса для логов и статуса.

After=network.target — порядок загрузки. Говорит systemd: «Не запускай этот сервис, пока полностью не поднимется сетевой стек (network.target)».

Секция [Service] (Логика работы и ограничения)
Type=simple — дефолтный тип. Означает, что процесс, запущенный в ExecStart, и есть главный процесс службы.

User=sasha — Безопасность! Запускает сервис от имени обычного пользователя sasha, а не от root.

WorkingDirectory — папка, из которой будет выполняться команда (где лежит наш index.html).

ExecStart — бинарник и точная абсолютная команда для запуска сервиса. (Важно: в systemd всегда указываются полные пути, например /usr/bin/python3, а не просто python).

Перезапуск при падении:
Restart=always — автоматически перезапускать процесс, если он завершился (по ошибке, аварийно или был убит сигналом).

RestartSec=3 — пауза в 3 секунды перед повторным запуском (чтобы сервис не зациклился в бесконечном мгновенном перезапуске, если в коде фатальная ошибка).

Ограничения ресурсов (cgroups v2):
Что такое cgroups (Control Groups): Это механизм ядра Linux, который позволяет ограничивать и изолировать ресурсы (CPU, память, ввод-вывод) для группы процессов. Systemd под капотом автоматически создает cgroup для каждого сервиса.

MemoryMax=100M — жесткий лимит памяти (Hard limit). Если сервис попытается сожрать больше 100 Мегабайт OOM-killer ядра Linux моментально убьет процесс.

MemoryHigh=80M — мягкий лимит (Soft limit). При превышении 80 МБ система начинает активно давить сервис, отбирая у него кеш и замедляя выделение памяти, но не убивает сразу.

Секция [Install] (Автозагрузка)
WantedBy=multi-user.target — указывает, при каком уровне загрузки системы включать сервис. multi-user.target — это стандартный режим работы Linux (многопользовательская консоль / обычная система с сетью).

Шаг 3. Запуск и активация сервиса
После создания или изменения любого unit-файла нужно принудительно заставить systemd перечитать конфигурации с диска:

Перечитываем конфиги systemd:

Bash
sudo systemctl daemon-reload
daemon-reload — заново сканирует директорию /etc/systemd/system/ и загружает новые правила в память.

Запускаем сервис:

Bash
sudo systemctl start myserver.service
Включаем автозапуск при старте системы:

Bash
sudo systemctl enable myserver.service
enable — создает симлинк (яpлык) в директории автозапуска systemd (/etc/systemd/system/multi-user.target.wants/).

Проверяем статус и лимиты:

Bash
systemctl status myserver.service
Вывод должен быть зеленого цвета (active (running)). В выводе ты также увидишь поддерево процессов, PID и текущее потребление памяти с учетом настроенных cgroups!

Шаг 4. Проверка устойчивости (Тест с kill -9)
Теперь проверим требование: поднимется ли сервис сам после жесткого аварийного завершения (SIGKILL).

Узнаем текущий PID (идентификатор) нашего запущенного Python-процесса. Его видно в systemctl status или можно найти командой:

Bash
pgrep -f "http.server 8080"
(Допустим, команда вернула PID 12345).

Имитируем жесткий сбой или фатальное падение системы, отправляя сигнал SIGKILL (-9):

Bash
sudo kill -9 12345
kill -9 — аварийное немедленное уничтожение процесса на уровне ядра. Процессу даже не дается времени на сохранение данных или корректное закрытие.

Ждем 3 секунды (наш RestartSec=3) и снова проверяем статус:

Bash
systemctl status myserver.service
Что ты увидишь в логах status:

Старый процесс с PID 12345 умер с кодом status=9/KILL.

Сработал Restart=always.

Systemd автоматически поднял новый процесс с НОВЫМ PID (например, 12399), и сервис снова работает как ни в чем не бывало!

Логи перезапуска в реальном времени также можно посмотреть системным архиватором логов:

Bash
journalctl -u myserver.service -n 20
journalctl — утилита просмотра логов systemd.

-u myserver.service (unit) — фильтр логов строго по нашему сервису.

-n 20 (number) — показать последние 20 строк.
