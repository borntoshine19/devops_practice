Давай настроим межсетевой экран (файрвол).

В Linux по умолчанию встроен модуль ядра **nftables** (или его предшественник **iptables**). Они невероятно мощные, но писать правила для них вручную — сложная и громоздкая задача.

Поэтому для Arch Linux идеально подходит **UFW (Uncomplicated Firewall)**.

### Что такое UFW и зачем он нужен?

- **Зачем нужен файрвол:** На сервере/ноутбуке может крутиться куча служб, открывающих сетевые порты (SSH, веб-серверы, Docker-контейнеры, базы данных). Без файрвола любой ПК в твоей локальной сети (или в интернете) может достучаться до этих портов. Файрвол создает фильтр: он **блокирует всё лишнее на входе** и пропускает только то, что ты явно разрешил (например, твой SSH).
    
- **Почему UFW:** Название переводится как «Неусложненный файрвол». Это удобный интерфейс (надстройка), который переводит твои простые человеческие команды вроде `ufw allow ssh` в сложные низкоуровневые правила ядра Linux.
    

> ⚠️ **КРИТИЧЕСКИ ВАЖНО:** Мы настраиваем файрвол по SSH. Если сначала включить файрвол, а только потом разрешать SSH, **тебя моментально выбросит из системы и заблокирует доступ**. Порядок команд ниже строго выверен.

### Шаг 1. Установка UFW

В Arch Linux UFW не идет в базовой системе, его нужно поставить из официальных репозиториев:

Bash

```
sudo pacman -S ufw
```

**Разбор команды:**

- `sudo` — запуск с правами суперпользователя (админа).
    
- `pacman` — стандартный менеджер пакетов в Arch Linux.
    
- `-S` (_Sync_) — флаг, который говорит менеджеру синхронизироваться с репозиторием и установить указанный пакет.
    
- `ufw` — имя устанавливаемой программы.
    

### Шаг 2. Задание правил по умолчанию (Дефолтная политика)

В сетевой безопасности есть золотое правило: **«Запрещено всё, что прямо не разрешено»**.

Выполняем две команды:

Bash

```
sudo ufw default deny incoming
```

**Разбор:** `default deny incoming` — устанавливает базовое правило: **блокировать любые входящие подключения** из сети к нашему ноутбуку, если для них нет отдельного разрешения.

Bash

```
sudo ufw default allow outgoing
```

**Разбор:** `default allow outgoing` — устанавливает противоположное правило: **разрешить любые исходящие подключения** с ноутбука во внешний мир. Это нужно, чтобы ты мог свободно выходить в интернет, скачивать пакеты через `pacman` и т.д.

### Шаг 3. Разрешаем доступ по SSH (Обязательный шаг!)

Теперь, когда входящий трафик по умолчанию заблокирован, нам нужно пробить «дыру» для нашего SSH-подключения, иначе после включения UFW мы потеряем связь.

Bash

```
sudo ufw allow 22/tcp
```

**Разбор команды и параметров:**

- `allow` — разрешить трафик.
    
- `22` — номер порта, на котором по умолчанию работает SSH-сервер.
    
- `/tcp` — протокол передачи данных (SSH работает строго поверх TCP).
    

_(Примечание: Можно написать просто `sudo ufw allow ssh`, UFW сам знает, что SSH — это 22/tcp)._

### Шаг 4. Включение файрвола и автозагрузки

Правила заданы. Теперь включаем сам файрвол:

Bash

```
sudo ufw enable
```

Система выведет предупреждение: `Command may disrupt existing ssh connections. Proceed with operation (y|n)?` _(Команда может оборвать текущие SSH-соединения)_. Поскольку мы **уже разрешили** порт 22 на Шаге 3, нажимаем **`y`** и `Enter`.

Теперь активируем автозапуск UFW при загрузке Arch Linux через `systemd`:

Bash

```
sudo systemctl enable ufw
```

**Разбор:** `systemctl enable` — добавляет службу `ufw` в автозагрузку системы, чтобы после перезагрузки ноутбука файрвол поднимался автоматически.

### Шаг 5. Проверка статуса

Проверим, как работают наши правила:

Bash

```
sudo ufw status verbose
```

**Разбор:**

- `status` — показать текущее состояние.
    
- `verbose` — вывести подробную информацию (включая дефолтные политики).
    

Вывод должен быть примерно таким:

Plaintext

```
Status: active
Logging: on (low)
Default: deny (incoming), allow (outgoing), disabled (routed)
New profiles: skip

To                         Action      From
--                         ------      ----
22/tcp                     ALLOW IN    Anywhere
```

Это значит:

1. `Status: active` — файрвол работает.
    
2. `Default: deny (incoming)` — все входящие соединения сбрасываются.
    
3. `22/tcp ALLOW IN Anywhere` — порт SSH открыт для всех.
    

Твой Arch Linux теперь защищен базовым файрволом!

Проверить, что файрвол реально работает и «наглухо» режет входящие соединения, можно двумя отличными способами: **быстрым** (сканированием портов с твоего ПК) и **наглядным** (через поднятие тестового сервиса и проверку логов UFW).

### Способ 1. Сканирование портов с помощью Nmap (Самый простой)

Идея простая: мы попробуем проверить с твоего ПК (Windows / MobaXterm), открыт ли на ноутбуке какой-нибудь порт, который мы **не разрешали** в UFW (например, порт `80` — обычный веб-сервер HTTP).

В MobaXterm или PowerShell на Windows выполни команду сканирования:

Bash

```
nmap -p 80,22 192.168.0.155
```

_(Если `nmap` не установлен в MobaXterm, его можно легко запустить через стандартную командную строку Windows / PowerShell, либо поставить утилиту `nmap`)._

**Разбор команды:**

- `nmap` (_Network Mapper_) — легендарный сканер сетевых портов и уязвимостей.
    
- `-p 80,22` — флаг **ports**. Мы просим проверить ровно два конкретных порта: `22` (наш SSH) и `80` (случайный неразрешенный порт).
    

**Что ты увидишь в ответе:**

Plaintext

```
PORT   STATE    SERVICE
22/tcp open     ssh
80/tcp filtered http
```

**Как читать результат:**

- `22/tcp open` — порт открыт и ответил на запрос (потому что мы прописали `ufw allow 22/tcp`).
    
- `80/tcp filtered` — статус **filtered** как раз и означает, что на ноутбуке работает **файрвол**, который проигнорировал и выбросил пакет в мусорку. Если бы файрвола не было, статус был бы `closed`.
    

### Способ 2. Поднять быстрый сервер и проверить с браузера (Практический)

Давай поднимем на ноутбуке временный веб-сервер на Python и попробуем зайти на него с компьютера через браузер.

#### 1. Запускаем тестовый сервер на ноутбуке

На ноутбуке в SSH-сессии выполни:

Bash

```
python -m http.server 8000
```

**Разбор:**

- `python -m http.server` — встроенный модуль Python, который мгновенно превращает текущую папку в веб-сервер.
    
- `8000` — порт, на котором сервер начинает слушать входящие подключения.
    

#### 2. Пробуем зайти с компьютера

Открой браузер на своем компьютере с Windows и введи в адресную строку: `[http://192.168.0.155:8000](http://192.168.0.155:8000)`

**Результат:** Страница будет бесконечно грузиться, а затем выдаст ошибку _«Не удается получить доступ к сайту»_ (таймаут). Это значит, что UFW блокирует пакеты еще на подлете и не дает им дойти до Python.

#### 3. Смотрим логи UFW в реальном времени

Открой **вторую** вкладку SSH на ноутбуке и выполни команду чтения системного журнала:

Bash

```
sudo dmesg -w | grep BLOCK
```

_(Или `sudo tail -f /var/log/errors.log` / `journalctl -f`)._

Ты увидишь, как UFW пишет в лог блокировки в реальном времени: `[UFW BLOCK] IN=wlan0 OUT= MAC=... SRC=192.168.0.X DST=192.168.0.155 PROTO=TCP SPT=... DPT=8000`

**Разбор строчки лога:**

- `[UFW BLOCK]` — событие блокировки пакета.
    
- `SRC=` — IP-адрес твоего компьютера (источник).
    
- `DST=` — IP-адрес ноутбука (назначение).
    
- `DPT=8000` — **Destination Port** (порт назначения). UFW зафиксировал попытку стукнуться на порт 8000 и заблокировал ее.
    

_(Чтобы остановить тестовый Python-сервер, нажми `Ctrl + C` в первом окне)._

- **Исходящее (Outgoing)** — это когда **твой Arch-ноутбук** сам проявляет инициативу и идет куда-то наружу (например, делает `pacman -Syu` в интернет или подключается к другому серверу).
    
- **Входящее (Incoming)** — это когда **кто-то снаружи** (в нашем случае твой браузер на Windows) проявляет инициативу и пытается подключиться к твоему Arch-ноутбуку.
  
  ### А как сделать так, чтобы зашло?

Чтобы Python-сервер на порту 8000 стал доступен для твоего ПК на Windows, тебе нужно явно разрешить этот входящий порт в UFW на ноутбуке:

Bash

```
sudo ufw allow 8000/tcp
```

Если после этой команды ты снова обновишь страницу `[http://192.168.0.155:8000](http://192.168.0.155:8000)` в браузере на Windows — всё мгновенно откроется, и ты увидишь список файлов!

А когда закончишь эксперимент, этот порт можно снова закрыть (удалить правило):

Bash

```
sudo ufw delete allow 8000/tcp
```

### Способ 1. Какие порты открыты в файрволе (UFW)

Эта команда показывает правила межсетевого экрана — какие порты разрешено пронимать из сети:

Bash

```
sudo ufw status verbose
```

**Разбор команды:**

- `status` — вывести список действующих правил.
    
- `verbose` — показать подробности (включая политики по умолчанию `default deny incoming`).
    

> **Как читать:** Все порты, которых **нет** в выводимом списке `ALLOW IN`, файрвол наглухо блокирует.

### Способ 2. Какие программы прямо сейчас «слушают» порты (ss)

Даже если порт открыт в UFW, на нем должна сидеть конкретная программа (например, SSH или Python). Команда `ss` показывает **все активные сетевые соединения и слушающие сервисы** в самой системе:

Bash

```
sudo ss -tulpn
```

**Разбор флагов (комбинация `tulpn`):**

- **`-t`** (_tcp_) — показывать **TCP**-порты.
    
- **`-u`** (_udp_) — показывать **UDP**-порты.
    
- **`-l`** (_listen_) — показывать только **слушающие** порты (тех, кто ждет входящих подключений, а не активные выходы в интернет).
    
- **`-p`** (_processes_) — показать **PID и имя программы**, которая заузила этот порт (требуются права `sudo`).
    
- **`-n`** (_numeric_) — выводить порты **числами** (например, `:22`), а не названиями служб (`:ssh`).
    

**Как читать вывод:** Ищи столбец `Local Address:Port` и `Users:(("process_name",...))`. Например: `LISTEN 0 128 0.0.0.0:22 ... users:(("sshd",pid=620,fd=3))` Это значит, что служба `sshd` слушает порт `22` на всех сетевых интерфейсах.

### Способ 3. Проверить конкретный порт одной строкой

Если тебе нужно быстро проверить, занят ли конкретный порт (например, `8000` или `22`), просто скомбинируй `ss` с фильтром `grep`:

Bash

```
sudo ss -tulpn | grep :22
```

**Разбор:**

- `|` (pipe / конвейер) — передает вывод первой команды на вход второй.

#### 2. Пробуем зайти с компьютера

Открой браузер на своем компьютере с Windows и введи в адресную строку: `[http://192.168.0.155:8000](http://192.168.0.155:8000)`

**Результат:** Страница будет бесконечно грузиться, а затем выдаст ошибку _«Не удается получить доступ к сайту»_ (таймаут). Это значит, что UFW блокирует пакеты еще на подлете и не дает им дойти до Python.

#### 3. Смотрим логи UFW в реальном времени

Открой **вторую** вкладку SSH на ноутбуке и выполни команду чтения системного журнала:

Bash

```
sudo dmesg -w | grep BLOCK
```

_(Или `sudo tail -f /var/log/errors.log` / `journalctl -f`)._

Ты увидишь, как UFW пишет в лог блокировки в реальном времени: `[UFW BLOCK] IN=wlan0 OUT= MAC=... SRC=192.168.0.X DST=192.168.0.155 PROTO=TCP SPT=... DPT=8000`

**Разбор строчки лога:**

- `[UFW BLOCK]` — событие блокировки пакета.
    
- `SRC=` — IP-адрес твоего компьютера (источник).
    
- `DST=` — IP-адрес ноутбука (назначение).
    
- `DPT=8000` — **Destination Port** (порт назначения). UFW зафиксировал попытку стукнуться на порт 8000 и заблокировал ее.
    

_(Чтобы остановить тестовый Python-сервер, нажми `Ctrl + C` в первом окне)._

- **Исходящее (Outgoing)** — это когда **твой Arch-ноутбук** сам проявляет инициативу и идет куда-то наружу (например, делает `pacman -Syu` в интернет или подключается к другому серверу).
    
- **Входящее (Incoming)** — это когда **кто-то снаружи** (в нашем случае твой браузер на Windows) проявляет инициативу и пытается подключиться к твоему Arch-ноутбуку.
  
  ### А как сделать так, чтобы зашло?

Чтобы Python-сервер на порту 8000 стал доступен для твоего ПК на Windows, тебе нужно явно разрешить этот входящий порт в UFW на ноутбуке:

Bash

```
sudo ufw allow 8000/tcp
```

Если после этой команды ты снова обновишь страницу `[http://192.168.0.155:8000](http://192.168.0.155:8000)` в браузере на Windows — всё мгновенно откроется, и ты увидишь список файлов!

А когда закончишь эксперимент, этот порт можно снова закрыть (удалить правило):

Bash

```
sudo ufw delete allow 8000/tcp
```

### Способ 1. Какие порты открыты в файрволе (UFW)

Эта команда показывает правила межсетевого экрана — какие порты разрешено пронимать из сети:

Bash

```
sudo ufw status verbose
```

**Разбор команды:**

- `status` — вывести список действующих правил.
    
- `verbose` — показать подробности (включая политики по умолчанию `default deny incoming`).
    

> **Как читать:** Все порты, которых **нет** в выводимом списке `ALLOW IN`, файрвол наглухо блокирует.

### Способ 2. Какие программы прямо сейчас «слушают» порты (ss)

Даже если порт открыт в UFW, на нем должна сидеть конкретная программа (например, SSH или Python). Команда `ss` показывает **все активные сетевые соединения и слушающие сервисы** в самой системе:

Bash

```
sudo ss -tulpn
```

**Разбор флагов (комбинация `tulpn`):**

- **`-t`** (_tcp_) — показывать **TCP**-порты.
    
- **`-u`** (_udp_) — показывать **UDP**-порты.
    
- **`-l`** (_listen_) — показывать только **слушающие** порты (тех, кто ждет входящих подключений, а не активные выходы в интернет).
    
- **`-p`** (_processes_) — показать **PID и имя программы**, которая заузила этот порт (требуются права `sudo`).
    
- **`-n`** (_numeric_) — выводить порты **числами** (например, `:22`), а не названиями служб (`:ssh`).
    

**Как читать вывод:** Ищи столбец `Local Address:Port` и `Users:(("process_name",...))`. Например: `LISTEN 0 128 0.0.0.0:22 ... users:(("sshd",pid=620,fd=3))` Это значит, что служба `sshd` слушает порт `22` на всех сетевых интерфейсах.

### Способ 3. Проверить конкретный порт одной строкой

Если тебе нужно быстро проверить, занят ли конкретный порт (например, `8000` или `22`), просто скомбинируй `ss` с фильтром `grep`:

Bash

```
sudo ss -tulpn | grep :22
```

**Разбор:**

- `|` (pipe / конвейер) — передает вывод первой команды на вход второй.
    
- `grep :22` — фильтрует текст и оставляет только строки, содержащие `:22`. Если команда ничего не вывела — порт свободен и никем не используется.
